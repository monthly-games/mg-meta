
# Monthly Games 저장소별 목표(Year 1)

> 목적: Monthly Games 조직의 GitHub 저장소별 **미션과 Year 1 목표(OKR 스타일)**를 정의해,  
> 사람과 AI가 같은 방향으로 움직이도록 기준을 제공한다.

---

## 0. 구조

각 저장소는 다음 3가지로 정의한다.

1. **Mission** : 이 레포가 존재하는 이유 (한 줄)
2. **Year 1 Objectives** : 1년 안에 달성하고 싶은 핵심 목표 (2~3개)
3. **Key Results (KRs)** : 달성 여부를 판단할 수 있는 정량/정성 기준

---

## 1. `mg-meta` – 비즈니스 & 프로세스 허브

### Mission
Monthly Games 사업의 **비전·전략·프로세스·템플릿을 버전 관리**하는 레포.

### Year 1 Objectives

- O1: “회사 룰북”을 코드처럼 관리할 수 있는 상태를 만든다.
- O2: 모든 프로젝트가 공통 템플릿을 사용하도록 기반을 구축한다.
- O3: AI가 참고할 수 있는 최소한의 Biz/Process 지식 베이스를 마련한다.

### Key Results

- KR1: 다음 3개 문서를 v1 상태로 완성하고 유지한다.
  - `docs/business/vision.md`
  - `docs/business/portfolio_strategy.md`
  - `docs/process/development.md`
- KR2: 템플릿 최소 3종을 운영에 활용한다.
  - `issue_spec_template.json`
  - `gdd_base.json`
  - `marketing_brief.json`
- KR3: Year 1 동안 생성되는 Epic 이슈의 90% 이상이  
  `issue_spec_template.json` 기반 `spec_json`을 포함한다.

## 2. `mg-common-game` – 공통 게임 엔진

### Mission
**한 번 만든 코어 엔진/시스템을 12개 게임이 재사용**하게 만들어,  
“한 달 1게임”이 기술적으로 가능하도록 하는 레포.

### Year 1 Objectives

- O1: 퍼즐 + 방치 + JRPG 공통 코어 엔진 v1을 안정화한다.
- O2: 3개 이상 게임이 동일한 엔진 버전을 공유하게 만든다.
- O3: AI가 수정하기 쉬운 모듈 구조와 테스트 기반을 만든다.

### Key Results

- KR1: 퍼즐/방치/JRPG 최소 모듈을 포함한 `v1.0.0` 릴리즈.
- KR2: Year 1 내 **최소 4개 게임**이 `engine_version >= 1.0.0` 사용.
- KR3: 전투/퍼즐/방치 핵심 모듈 테스트 커버리지 **50% 이상**.
- KR4: 공통 엔진 관련 버그 중 “엔진 자체 문제” 비율 **20% 이하** 유지.

## 3. `mg-common-backend` – 공통 백엔드

### Mission
인증, 경제, 가챠, 텔레메트리 같은 **반복 백엔드 기능을 한 번만 구현**하고,  
모든 게임이 공유하게 만드는 레포.

### Year 1 Objectives

- O1: 공통 계정/경제/가챠 백엔드 v1을 제공한다.
- O2: 서버 코드 변경 없이 게임별 밸런스/설정을 Firestore/Remote Config로 제어 가능하게 한다.
- O3: 운영/치트 리스크를 최소화하는 기본 방어선을 구축한다.

### Key Results

- KR1: 계정/경제/가챠/로그 수집 포함 Functions v1 운영.
- KR2: **최소 3개 게임**이 공통 백엔드만으로 런칭 (게임 전용 API 없이).
- KR3: 결제/경제 관련 크리티컬 버그(롤백 필수급) **0건** 유지.
- KR4: 치트/어뷰징 탐지 로직으로 적발되는 의심 트래픽 비율 **1% 이상** 확보.

## 4. `mg-common-analytics` – 공통 분석

### Mission
모든 게임의 데이터를 **하나의 이벤트/스키마/쿼리 체계**로 분석하게 하는 레포.

### Year 1 Objectives

- O1: 공통 이벤트 스키마 v1을 정의하고 적용한다.
- O2: 핵심 KPI(DAU, Retention, ARPDAU, LTV)를 단일 뷰에서 제공한다.
- O3: 게임별 “기본 지표 대시보드 생성 비용”을 거의 0에 가깝게 낮춘다.

### Key Results

- KR1: `events.json` v1 확정 + Year 1 게임 중 **최소 6개**가 이 스키마 사용.
- KR2: `fact_user_daily`, `fact_revenue` 뷰 운영  
  (모든 게임 데이터가 이 뷰에서 조회 가능).
- KR3: 신규 게임 추가 시 **추가 SQL 작성 없이** 핵심 KPI 조회 가능.
- KR4: 지표 정의/불일치 관련 이슈를 분기당 **1건 이하**로 유지.

## 5. `mg-common-infra` – 인프라

### Mission
GCP/Firebase/BigQuery 인프라를 **Terraform으로 완전히 코드화**해,  
“누가 배포해도 같은 결과”가 나오게 하는 레포.

### Year 1 Objectives

- O1: dev/staging/prod 3환경 템플릿을 완성하고 실제 운영에 쓴다.
- O2: 새 게임 추가 시 Terraform 코드만으로 인프라를 준비할 수 있게 한다.
- O3: Infra 변경은 100% PR + plan 기반으로만 수행하게 만든다.

### Key Results

- KR1: `env/dev`, `env/staging`, `env/prod` 모든 환경을 실제로 사용.
- KR2: 새 게임 인프라 셋업에 필요한 수동 GCP 콘솔 작업 **0건**.
- KR3: Year 1 동안 `terraform apply`는 **100% PR 기반**으로 수행.
- KR4: 인프라 설정/권한 문제로 인한 장애 **3건 이하** 유지.

---

## 6. `mg-common-marketing` – 마케팅

### Mission
UA/스토어/크리에이티브를 **JSON 템플릿으로 제품화**해,  
마케팅도 “복붙 + 약간 수정”만으로 돌아가게 하는 레포.

### Year 1 Objectives

- O1: 퍼즐/방치/JRPG 공통 마케팅 템플릿 v1 세트를 완성한다.
- O2: 스토어 설명/기본 광고 카피를 템플릿에서 바로 뽑을 수 있게 한다.
- O3: 게임 1개당 마케팅 세팅 시간을 1일 이내로 줄인다.

### Key Results

- KR1: 장르별(퍼즐/방치/JRPG) 캠페인 템플릿 최소 1개씩 운영.
- KR2: Year 1 게임 중 **최소 6개**가 템플릿 기반으로 스토어 설명 생성·수정.
- KR3: 새 게임 런칭 시,  
  캠페인/크리에이티브 JSON 작성 + 세팅 작업 시간이 **8시간 미만**.
- KR4: 광고 카피/스토어 설명을 AI로 “처음부터 새로 작성”하는 비율 **20% 이하**.

## 7. 게임 저장소 공통 목표 – `mg-game-000X`

### 공통 Mission
공통 엔진/백엔드/인프라/분석/마케팅 템플릿을 활용해  
**“한 달 1게임”을 실제 릴리즈 단위로 구현**하는 레포.

### 공통 Year 1 Objectives

- O1: 각 게임당 1회 이상 실제 스토어/실사용자에게 노출한다.
- O2: 공통 모듈 재사용 비율을 극대화한다.
- O3: 개발/운영 비용을 줄이면서도, 실험/학습은 꾸준히 쌓는다.

### 공통 Key Results

- KR1: 각 게임 `v1.0.0` 빌드 최소 1회 릴리즈 (Soft 또는 Small Launch 가능).
- KR2: 각 게임의 코드 기준,  
  `common/` 모듈 의존 비율(라인수 기준) **50% 이상**.
- KR3: 각 게임 당 “기본 지표 세트(D1/D7, 튜토리얼 퍼널, ARPU)”를  
  BigQuery에서 **1개 쿼리로 조회 가능**.
- KR4: Year 1 12개 게임 중 **최소 3개**는  
  2개월 이상 라이브 서비스 유지 & 월간 업데이트 1회 이상.

## 10. 활용 방법

1. 이 문서를 `mg-meta/docs/business/repo_objectives.md`로 저장한다.
2. 각 저장소의 `README.md`에 “이 저장소의 Year 1 목표” 링크를 추가한다.
3. 분기별 회고/계획 회의 때:
   - 각 저장소별 Objectives/KRs 달성률을 간단 체크리스트로 리뷰한다.
4. AI에게 태스크/설계를 맡길 때:
   - 이 문서의 해당 저장소 Mission/Objectives를 먼저 참고하도록 지시한다.
