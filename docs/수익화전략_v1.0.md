# Monthly Games 수익화 전략 문서 v1.0
- 작성일: 2025-12-12
- 전제: **월 1개 게임 출시(2D 모바일)** / Flutter·Firebase·GCP / GA4+BigQuery 분석 / 자동화 중심 운영

---

## 1) 회사 아이덴티티(Identity) 정의
### 1.1 우리가 파는 것
- “게임 1개”가 아니라 **반복 가능한 제작·운영 시스템**과 **검증된 재미 루프**를 판다.
- 유저에게는: *짧게 시작해도 몰입되는 경험 + 부담 없는 결제 경험 + 업데이트로 점점 좋아지는 게임*

### 1.2 수익화 철학(비협상 원칙)
- Pay-to-Win(승패를 돈으로 결정) 금지
- Pay-for-Style(외형) / Pay-for-Speed(시간 단축) 허용
- **광고는 강제하지 않고 선택권 중심(Rewarded-first)**
- 유저 신뢰(환불/표기/확률/가격 투명성) > 단기 매출
- “한 번의 대박”이 아니라 **30일 LTV를 안정적으로 쌓는 구조**를 표준화

## 2) Monthly Games에 최적화된 ‘포트폴리오 수익화’ 전략
월 1개 출시 구조에서는 **단일 게임의 과금 모델**보다,
“장르별로 가장 성공 확률 높은 모델을 표준화”하고 **데이터로 빠르게 보정**하는 것이 핵심이다.

### 2.1 장르별 기본 수익화 모드(권장 기본값)
| 장르 | 권장 모드 | 이유 | 핵심 축 |
|---|---|---|---|
| 퍼즐 | **Hybrid(무료 진입 + 유료 언락 + 선택 광고)** | 유료 전환이 상대적으로 쉬움, 몰입 보존 | Premium Unlock + Rewarded |
| 방치형 | **F2P 표준(배틀패스/번들/광고/부스트)** | 장기 플레이·반복 루프·LiveOps에 최적 | Pass + Bundles + Boost |
| JRPG 요소 | **F2P 표준(가챠/스킨/패스) + DLC 가능** | 수집·성장·캐릭터 욕구 | Gacha + Cosmetics + Pass |

### 2.2 “공통 모듈” 표준 6종(모든 게임 탑재)
1) Rewarded Ads 모듈(선택 광고)  
2) 번들/오퍼 모듈(Starter/Value/Comeback)  
3) Premium Unlock(유료 언락/광고 제거 옵션)  
4) 배틀패스(월간 28일)  
5) 코스튬/스킨 샵(외형 중심)  
6) 개인화(세그먼트 기반: 광고/오퍼/난이도/보상 계수)

> 각 게임은 “모듈 ON/OFF + 파라미터”만 조정한다.  
> 구현은 공통, 차이는 **구성 값(config)** 으로 만든다.

## 3) 표준 수익 구조 설계(모듈별 정책)
### 3.1 광고(Rewarded First) 정책
- 기본: 보상형 광고(Rewarded) 중심
- Interstitial은 “피로도 높은 포인트”에만 제한적으로 사용(기본 OFF)
- 유저별 개인화:
  - 광고 선호(Ad Affinity) 높음 → 보상형 빈도 ↑, 보상 계수 = 기본
  - 광고 선호 낮음 → 빈도 ↓, 보상 계수 ↑, Premium Unlock 제안 ↑

**가드레일**
- Rewarded: 10회/일(기본), cooldown 120초(기본)
- Interstitial: 0~3회/일(기본), 세션 첫 3분 금지

### 3.2 Premium Unlock(유료 언락) 정책
- 퍼즐/캐주얼에 최적
- 구조:
  - 무료 구간: 30~60분 플레이 가능(또는 챕터 1)
  - 이후 선택:
    - (A) ₩3,900~₩7,900 언락
    - (B) 광고로 계속(느리게)
- 언락 포함 옵션:
  - 광고 제거
  - 일일 보너스(소량 Hard currency)
  - 추가 스테이지(또는 보너스 챕터)

### 3.3 번들/오퍼(카탈로그 표준)
| 번들 | 가격대 | 노출 조건 | 목적 |
|---|---:|---|---|
| Starter Pack | $0.99 | D0~D3, 첫 성취 직후 | 첫 결제 전환(Conversion) |
| Value Bundle | $4.99~$9.99 | 진행 정체/주요 성장 병목 | ARPPU 상승 |
| Comeback Pack | $2.99 | 7일 이상 미접속 후 복귀 | 재활성화 |
| Cosmetic Drop | $1.99~$4.99 | 선호 캐릭터/스킨 추천 | 신뢰 기반 매출 |

**가드레일**
- 강제 팝업 최소화: 1세션 1회 이하
- 구매 유도는 “도움”의 형태로, 결제 없이는 진행 불가 구조 금지

### 3.4 배틀패스(월간 28일 표준)
- 월 1게임 구조와 동일한 리듬(28일)로 표준화
- 무료 트랙: 잔존 유지
- 유료 트랙: 고가치(스킨 1 + 성장부스트 + 재화)
- 목표: ARPDAU 안정화 + 업데이트 루틴 강제

### 3.5 가챠(필요한 게임만 ON)
- 방치/JRPG 성격 강한 게임만 사용
- 원칙:
  - 천장(pity) 고정
  - 확률 표기/기록(로그) 필수
  - 스킨/연출로 가치를 느끼게 하고, 성능 차이는 제한(메타 붕괴 방지)

## 4) 개인화 기반 수익화(핵심 경쟁력)
### 4.1 개인화 목표
- 같은 게임이라도 유저별 최적 경로를 제공:
  - 광고형(Rewarded) 수익
  - Premium Unlock 수익
  - 번들/패스 수익

### 4.2 세그먼트(권장 4분류)
- F2P-Engaged(광고·반복 플레이 높음)
- F2P-Low(관심 낮음, 이탈 위험)
- Payer-Low(소액 결제 가능)
- Payer-High(가치형 번들 반응)

### 4.3 개인화 적용 영역(우선순위)
1) 오퍼 타입 선택(Starter/Value/Comeback)  
2) 광고 빈도/보상 계수(Ad Frequency / Reward Factor)  
3) Premium Unlock 타이밍/가격(할인 포함)  
4) 난이도/힌트(이탈 방지용)

## 5) 30일 수익화 런칭 플랜(표준 운영 캘린더)
### D0~D1 (첫 인상)
- 목표: 튜토리얼 완주율 + 첫날 재방문
- 실행:
  - Rewarded 1~2개 placement만 노출(과노출 금지)
  - Starter Pack: “첫 성취(첫 보스/첫 챕터 완료)” 직후 1회만 제안
  - Premium Unlock(퍼즐): 챕터 1 종료 시점에서 제안(선택권 강조)

### D2~D7 (전환/습관)
- 목표: 첫 결제 전환 + 광고 습관 형성 + 패스 진입
- 실행:
  - Battle Pass 노출(28일) + 목표 달성 보상(무료 트랙 강함)
  - Value Bundle은 ‘진행 정체’ 시점에서만 노출(개인화)
  - Comeback 트리거 준비(3일 이탈부터)

### D8~D30 (LTV 안정화)
- 목표: 이벤트/패스/코스튬으로 매출 안정화
- 실행:
  - 주간 이벤트(1) + 주간 코스튬 드랍(1)
  - 고과금 후보(Payer-High)에게 가치형 번들(고가치/저빈도)
  - 이탈 위험군에게 “보상형 이벤트 + 난이도 완화” 제공

## 6) KPI & 의사결정 기준(자동 최적화용)
### 6.1 런칭 KPI(기본 목표값 예시)
- D1: 30~40%
- D7: 12~18%
- 구매 전환율: 0.6~1.2%
- Ads ARPDAU: $0.05~$0.15
- Total ARPDAU: $0.15~$0.35

### 6.2 자동 최적화 룰(가드레일 포함)
- D1 낮음 → 튜토리얼/힌트 강화, 초기 보상 +10~20%
- Ads ARPDAU 낮음 → Rewarded placement 개선(보상/UX), 빈도는 과격하게 올리지 않음
- 구매 전환율 낮음 → Starter Pack 가치 재설계(가치↑, 노출↓), 가격 실험
- P2W 리스크 → 스탯 격차 상한 설정(예: ±25%)
- 불만/환불 증가 → 팝업 축소, 확률/가격/가치 표기 강화

## 7) 데이터·분석 설계(실행 가능 수준)
### 7.1 GA4 이벤트(최소 필수)
- session_start, tutorial_complete
- stage_start, stage_clear, stage_fail
- rewarded_ad_available, rewarded_ad_view, rewarded_ad_reward
- offer_impression, offer_click, purchase
- premium_unlock_offer, premium_unlock_purchase

### 7.2 BigQuery 집계(일 1회)

- monetization_config_version(date, version, changes)

## 8) 운영 거버넌스(월 1개 출시 최적)
### 8.1 구성 값(Config) 기반 운영
- 수익화는 **코드 수정 없이 config로** 변경
- 저장 위치:
  - Remote Config: 빠른 토글/간단 파라미터
  - Firestore: 복잡한 카탈로그/오퍼/세그먼트 규칙
- 변경 이력 필수(버전, 변경 사유, 영향 범위)

### 8.2 A/B 테스트 원칙
- 월 1게임 구조에서는 “테스트 남발” 금지
- 한 번에 1~2개 가설만
- 7일 단위로 평가(또는 트래픽 충분 시 조기 종료)
